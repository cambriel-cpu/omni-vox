## 1. Gateway "Pairing Required" Issue

### The Problem
When I (the agent) try to use `gateway` tools like `config.apply`, `config.patch`, or `restart`, I get:
```
gateway closed (1008): pairing required
Gateway target: ws://172.17.0.15:18789
Source: local lan 172.17.0.15
```

### Root Cause
I'm running inside a **Docker sandbox container**. The sandbox connects to the gateway via WebSocket from the Docker network IP (172.17.0.x), not from localhost (127.0.0.1). Per OpenClaw's security model:
- **Local connections (127.0.0.1) are auto-approved** for device pairing
- **Non-local connections (LAN, Tailnet, Docker bridge)** require explicit device pairing approval

The gateway is bound to `lan` (0.0.0.0), which lets the sandbox reach it — but since the connection comes from 172.17.0.x, it's treated as a remote device needing pairing.

### Fixes (in order of recommendation)

**Option A: Approve the sandbox as a device**
```bash
openclaw devices list          # see pending pairing requests
openclaw devices approve <id>  # approve the sandbox device
```
This should be a one-time approval. The sandbox container gets a stable device ID.

**Option B: Enable `commands.restart` for chat-based restart**
In `openclaw.json`, set:
```json
"commands": { "restart": true }
```
Then I can use `/restart` as a chat command instead of the gateway tool.

**Option C: Add Docker network to trusted proxies**
If OpenClaw supports `gateway.trustedProxies`, adding the Docker bridge subnet (172.17.0.0/16) would let the gateway recognize sandbox connections. (Need to verify this exists.)

**Option D: Use `config.patch` via CLI**
For config changes, Chris can run `openclaw config set` from the host CLI, or use the Control UI config editor — both bypass the sandbox pairing issue.

### Remote Access (Away from Home)
For managing OpenClaw remotely without LAN access:
- **Tailscale** is recommended: set `gateway.bind: "tailnet"` and use Tailscale to access the gateway from anywhere
- **Control UI** works over Tailscale too
- The current `gateway.bind: "lan"` only works on the local network
- Consider setting up Tailscale on the Omnissiah if not already done
